Standalone Deployment for Excell

The Excell ecosystem can also be deployed on a Standalone VM setup, where services are installed and managed directly on virtual machines (VMs) without container orchestration. This deployment model is designed for environments where Kubernetes is not feasible or preferred, allowing services to run as system processes while ensuring efficient resource utilization.

Note: The Keycloak and PostgreSQL setup is handled separately and is not included in this deployment plan.

Standalone Deployment Overview

The Excell application stack consists of the following components:

  • Databases: MongoDB, Redis, Neo4j
  • Products: Fluent Bit, Prometheus, Grafana, Nginx
  • Microservices: Core API, Keycloak Admin API, Notification API, Storage API, External API, Observability Service

Unlike Kubernetes, where workloads are dynamically distributed across nodes, a Standalone VM deployment follows a structured assignment of services to specific VMs to ensure resource efficiency and maintainability.

Infrastructure Setup

1. Database & Caching Node

Component Description
MongoDB Handles document-based storage for various microservices.
Redis Acts as a caching layer to improve response times.
Neo4j Graph database for managing network infrastructure inventory.
  • Number of VMs: 2
  • CPU & RAM Estimate: 8 vCPU, 32GB RAM each

2. Microservices Nodes

Microservices Description
Core API Manages business logic and application workflows.
Keycloak Admin API Handles IAM-related admin operations.
Notification API Processes and sends notifications via SMS, Email, and Mobile Push.
Storage API Provides file storage operations with access control.
External API Interfaces with third-party services such as Aadhaar authentication.
Observability Service Manages logging, metrics, and system health monitoring.
  • Number of VMs: 3
  • CPU & RAM Estimate: 8 vCPU, 32GB RAM each

3. Observability & Monitoring Node

Observability Components Description
Fluent Bit Collects and forwards logs from all services.
Prometheus Monitors and scrapes system and application metrics.
Grafana Provides dashboards for real-time system insights.
  • Number of VMs: 1
  • CPU & RAM Estimate: 4 vCPU, 16GB RAM

4. Load Balancer & Reverse Proxy Node

Component Description
Nginx Handles incoming traffic, routes requests to the appropriate services, and provides SSL termination.
  • Number of VMs: 1
  • CPU & RAM Estimate: 4 vCPU, 16GB RAM

Deployment Considerations

Service Distribution Strategy

In a standalone deployment, each service runs as a system process on its designated VM.

  • Systemd or Supervisor will be used to ensure services automatically restart in case of failure.
  • Manual scaling is required by provisioning additional VMs when needed.

Storage and Persistence

  • MongoDB and Neo4j will store data on high-performance SSD-backed volumes.
  • Redis will be configured with snapshotting and AOF (Append-Only File) for durability.

Security & Networking

  • Firewall rules will be configured to restrict unnecessary external access.
  • TLS encryption will be enabled for API communication.
  • RBAC-based access control will be enforced using Keycloak authentication mechanisms.

Minimum Standalone VM Allocation

VM Group No. of VMs CPU Allocation RAM Allocation
Database Nodes 2 8 vCPU each 32GB RAM each
Microservices Nodes 3 8 vCPU each 32GB RAM each
Observability Node 1 4 vCPU 16GB RAM
Load Balancer Node 1 4 vCPU 16GB RAM
Total 7 VMs ~52 vCPU ~224GB RAM

This Standalone VM deployment ensures a structured, controlled, and optimized resource allocation while maintaining operational simplicity for environments that do not require container orchestration.
